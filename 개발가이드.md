## API 설계

로그인 사용자 전용. 비로그인은 localStorage로만 동작.

| Method | URL | 설명 |
|--------|-----|------|
| `GET` | `/api/v1/checklists/{type}/progress` | 완료된 item_id 목록 조회 |
| `PUT` | `/api/v1/checklists/{type}/items/{itemId}` | 항목 완료/미완료 토글 |
| `DELETE` | `/api/v1/checklists/{type}/progress` | 체크리스트 초기화 |

- `{type}`: `new-home` / `jeonse-move` / `monthly-move`
- 인증 헤더: `Authorization: Bearer {token}`
- progress는 첫 항목 체크 시 자동 생성 (별도 초기화 API 불필요)

### 응답 예시 (`GET /api/v1/checklists/new-home/progress`)
```json
{
  "completedItemIds": ["realtor-verification", "registry-check", "building-ledger"]
}
```

> 체크리스트 구조(섹션, 항목 텍스트)는 응답에 포함하지 않음. 프론트 상수에서 조립.

## 코드 패턴

### 예외 처리
```java
// 커스텀 예외 정의
public class ChecklistProgressNotFoundException extends RuntimeException {
    public ChecklistProgressNotFoundException() {
        super("체크리스트 진행 상태를 찾을 수 없습니다.");
    }
}

// 전역 핸들러에서 처리
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(ChecklistProgressNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(ChecklistProgressNotFoundException e) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(new ErrorResponse(e.getMessage()));
    }
}
```

## 개발 원칙
- 레이어별 역할 분리: 컨트롤러는 HTTP 요청/응답만. 서비스에 비즈니스 로직
- 메서드별 역할 분리: 하나의 메서드는 하나의 책임만
- 삼항 연산자 2개 이상 이어서 사용하지 않기
- 테스트 코드 작성하기
- 비즈니스 로직 예외는 커스텀 예외 정의. 예외 핸들러 분리
