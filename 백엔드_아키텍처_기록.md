# Backend — 아키텍처 결정 기록

## 전체 구조

- 레이어드 아키텍처 + 선택적 헥사고날 원칙
- 외부 API처럼 구현이 바뀔 수 있는 지점에만 Port/Adapter 패턴을 적용. 나머지는 일반 레이어드 방식

```
core/                   도메인 + 비즈니스 규칙 (외부 의존 없음)
├── domain/
├── port/               core가 외부에 요구하는 계약 (interface)
├── repository/         Spring Data JPA (포트로 감싸지 않음)
└── service/            비즈니스 로직

api/                    실행 모듈
├── controller/         HTTP 진입점
├── infrastructure/     Port 구현체 (외부 API 어댑터)
│   ├── bldg/           건축물대장 HTTP 클라이언트
│   └── molit/          국토부 실거래가 HTTP 클라이언트
└── service/            인증 등 api 계층 전용 서비스
```

## 포트 설계 고민: Actor 기준 vs 기능 기준

### 문제

전용면적을 구하는 방법은 달라질 수 있음. 현재는 두 가지 소스를 사용
1. 건축물대장 API (1순위)
2. 국토부 실거래가 API (2순위 fallback)

초기에는 전용면적용 포트 를 두고 전용면적 어댑터에서 Client 를 조합하는 방식

### 문제점
- "건축물대장 먼저 조회 후 실패시 실거래가 조회"라는 우선순위 정책은 **비즈니스 결정**  
- 비즈니스 정책이 인프라 계층인 어댑터 안에 있으면, 정책이 바뀔 때 어댑터를 건드려야 하는 문제 있음  
- 어댑터는 "어떻게 API를 호출하는가"만 알아야 하지, "어디서 먼저 가져올 것인가"를 결정해서는 안됨

### Actor 기준 포트를 고려했지만 기각

```
// core가 외부 시스템 이름을 알게 됨
private final BldgLedgerPort bldgLedgerPort;  // BldgLedger가 뭔지 core가 알아야 함
private final MolitPort molitPort;
```

- Actor(데이터 소스) 이름을 포트에 쓰면 BldgLedger를 다른 API로 교체할 때 포트 이름까지 바꿔야함  

### 채택한 방식: 기능 기준 포트 + 비즈니스 로직을 서비스로

```
core/port/
├── BuildingRegistryPort   "건물 등록 정보가 필요해"  (BldgLedger 몰라도 됨)
└── TradeDataPort          "거래 데이터가 필요해"     (MOLIT 몰라도 됨)

core/service/
└── ExclusiveAreaService   fallback 우선순위 결정 (비즈니스 정책)

api/infrastructure/
├── BldgLedgerAdapter      BuildingRegistryPort 구현
└── MolitTradeAdapter      TradeDataPort 구현
```

- 건물 등록 정보를 가져오는 방식이 변경되면, `BuildingRegistryPort` 를 구현하는 다른 어댑터로 교체
- 비즈니스 로직이 변경되면 `ExclusiveAreaService`에서 변경

## 컨트롤러에서 포트 직접 참조

### 현황

`DiagnosisController.getTransactions()`는 `MarketPricePort`를 직접 주입받아 사용한다.

### 판단
`MarketPricePort`는 현재 비즈니스 로직이 없는 단순 조회여서, 서비스를 만들면 위임만 하는 보일러플레이트가 됨.

**결론:** 현재는 포트 직접 참조 허용. `MarketPricePort`에 정책이 생기는 시점에 `MarketPriceService` 사용하도록 수정
