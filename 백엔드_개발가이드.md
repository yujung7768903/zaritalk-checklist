# 개발 원칙

## 아키텍처(레이어드 아키텍처 + 헥사고날 일부 원칙)
### 배경
- 규모가 크지 않으므로 레이어드 아키텍처를 사용하되 비즈니스 로직의 기술 의존도를 줄이고 쉬운 테스트를 위해 헥사고날 일부 원칙을 함께 사용한다.
### 원칙
#### 외부 시스템은 Port로 추상화한다(예시:국토교통부, 건축물대장과 같은 외부 API) 
 예시
```java
public interface PaymentPort {
void pay(Payment payment);
}
```
```java
@Service
public class PaymentService {

    private final PaymentPort paymentPort;

    public void pay() {
        paymentPort.pay(payment);
    }
}
```
```java
@Component
public class TossPaymentAdapter implements PaymentPort {
}
```

#### 기능 단위로 책임을 분리한다.
예시
```
PlanCommandService
PlanQueryService
```

#### Repository는 port로 구현하지 않는다
- 각 테이블에서 데이터를 조회할 때는 해당 테이블에 대한 Repository 하나만 필요하고, Port로 구현할 경우 JPA에서 기본적으로 제공하는 풍부한 기능을 감추게 되기 때문에 port로 구현하지 않는다.

## 관심사 분리
### 배경
- 유지보수와 가독성을 고려하여 관심사가 다른 코드는 분리한다.
### 원칙
- 변경을 고려했을 때 변경 시점이나 이유가 다르다면 관심사가 다른 것으로 간주한다.
### 예시
나쁜 예시
```java
public class PaymentService {
    public Payment prepare(Long orderId, String currency, BigDecimal foreignCurrencyAmount) throws IOException {
        // 환율 가져오기
        URL url = new URL("https://open.er-api.com/v6/latest/" + currency);
        HttpURLConnection connection = (HttpURLConnection) url.openConnection();
        BufferedReader br = new BufferedReader(new InputStreamReader(connection.getInputStream()));
        String response = br.lines().collect(Collectors.joining());
        br.close();

        ObjectMapper mapper = new ObjectMapper();
        ExRateData data = mapper.readValue(response, ExRateData.class);
        System.out.println(data);
        BigDecimal exRate = data.rates().get("KRW");

        // 금액 계산
        BigDecimal convertedAmount = foreignCurrencyAmount.multiply(exRate);

        // 유효 시간 계산
        LocalDateTime validUntil = LocalDateTime.now().plusMinutes(30);

        return new Payment(orderId, currency, foreignCurrencyAmount, BigDecimal.ZERO, BigDecimal.ZERO, validUntil);
    }
}
```
좋은 예시
```java
public class PaymentService {
    private final ExRateProvider exRateProvider;
    private final Clock clock;

    public PaymentService(ExRateProvider exRateProvider, Clock clock) {
        this.exRateProvider = exRateProvider;
        this.clock = clock;
    }

    public Payment prepare(Long orderId, String currency, BigDecimal foreignCurrencyAmount) {
        BigDecimal exRate = exRateProvider.getExRate(currency);

        return Payment.createPrepared(orderId, currency, foreignCurrencyAmount, exRate, LocalDateTime.now(clock));
    }
}
```

## 매직 넘버
### 배경
- 의도를 파악하기 어려워 가독성이 저하된다.
- 동일한 숫자를 여러 곳에서 사용할 때 수정 범위를 파악하기 어렵다.
### 원칙
- 의미를 나타내는 상수로 선언하여 사용한다.

## 디미터 법칙
### 배경
- 객체 간의 결합도를 낮추고 유지보수를 쉽게 만들기 위해 해당 법칙을 적용한다.
### 원칙
- 이웃 객체만 알며, 타 객체의 내부 속사정의 몰라야한다.
- (. 연쇄 사용): a.getB().getC().doSomething() 과 같이 다른 객체의 내부 구조를 파고들어 연쇄적으로 접근하는 것은 객체의 캡슐화를 파괴하므로 지양해야합니다.

## 객체 생성 책임

### 배경
- 생성 규칙이 여러 곳에 흩어지면 객체 생성 방식이 일관되지 않다.
- 객체 생성 시 검증 로직이 누락될 수 있다.

### 원칙
- 객체 생성 규칙은 객체 내부 또는 전용 Factory에서 관리한다.
- 생성자를 직접 호출하는 대신 의미 있는 생성 메서드를 제공한다.

### 예시

나쁜 예시

```java
Payment payment = new Payment(
    orderId,
    currency,
    amount,
    exchangeRate,
    validUntil
);
```

좋은 예시

```java
Payment payment = Payment.createPrepared(
    orderId,
    currency,
    amount,
    exchangeRate,
    validUntil
);
```

## Null 대신 명시적인 표현 사용

### 배경
- NullPointerException 발생 가능성이 높아진다.
- 호출자가 null 여부를 항상 고려해야 한다.

### 원칙
- Optional은 반환값에 한해서 사용한다.
- 빈 컬렉션을 반환할 수 있다면 null 대신 빈 컬렉션을 반환한다.
- 메서드 인자나 필드로 Optional을 사용하지 않는다.

### 예시

나쁜 예시

```java
return null;
```

좋은 예시

```java
return Optional.empty();
```

```java
return Collections.emptyList();
```

## 조기 반환(Early Return)

### 배경
- 중첩 if 문은 가독성을 떨어뜨린다.
- 핵심 비즈니스 로직이 깊은 블록 안에 숨어버린다.

### 원칙
- 예외 조건은 먼저 처리하고 반환한다.
- 정상 흐름(Happy Path)을 아래에 배치한다.

### 예시

나쁜 예시

```java
if (user != null) {
    if (user.isActive()) {
        process();
    }
}
```

좋은 예시

```java
if (user == null) {
    return;
}

if (!user.isActive()) {
    return;
}

process();
```

### 배경
- 상태 변경은 버그를 유발하기 쉽다.
- 객체의 생성 시점과 변경 시점을 추적하기 어렵다.

### 원칙
- 가능하면 final을 사용한다.
- DTO는 변경 불가능하게 설계한다.
- 컬렉션은 외부에서 수정할 수 없도록 반환한다.

### 예시

```java
public record UserResponse(
    Long id,
    String name
) {
}
```

## Enum 활용

### 배경
- 문자열 비교는 오타에 취약하다.
- 상태값이 분산되기 쉽다.

### 원칙
- 의미 있는 상태값은 Enum으로 관리한다.
- 문자열 상수로 상태를 표현하지 않는다.

### 예시

나쁜 예시

```java
if ("ACTIVE".equals(status))
```

좋은 예시

```java
if (UserStatus.ACTIVE == status)
```

## 비즈니스 규칙은 Service 또는 Domain에 위치

### 배경
- Controller에 비즈니스 로직이 들어가면 재사용이 어렵다.
- Repository에 비즈니스 로직이 들어가면 책임이 모호해진다.

### 원칙
- Controller는 요청/응답 처리만 담당한다.
- Repository는 데이터 접근만 담당한다.
- 비즈니스 규칙은 Service 또는 Domain에 위치시킨다.

### 예시

나쁜 예시

```java
@GetMapping
public ResponseEntity<?> create() {

    if (user.getAge() < 19) {
        throw new Exception();
    }

}
```

좋은 예시

```java
planService.createPlan(userId);
```

## 엔티티 직접 노출 금지

### 배경
- API 스펙이 DB 구조에 종속된다.
- Entity 변경이 API 변경으로 이어질 수 있다.
- 순환 참조 및 불필요한 데이터 노출 문제가 발생할 수 있다.

### 원칙
- Controller와 외부 계층 간에는 DTO를 사용한다.
- Entity는 Persistence Layer 내부에서만 사용한다.

### 예시

나쁜 예시

```java
@GetMapping
public User getUser()
```

좋은 예시

```java
@GetMapping
public UserResponse getUser()
```

## 로그 작성 원칙

### 배경
- 운영 환경에서 장애 원인을 추적하기 위해 필요하다.
- 과도한 로그는 오히려 문제 분석을 어렵게 만든다.

### 원칙
- 주요 비즈니스 이벤트는 INFO 레벨로 기록한다.
- 예외 상황은 ERROR 레벨로 기록한다.
- 개인정보 및 민감정보는 로그에 남기지 않는다.
- 로그 메시지에는 추적 가능한 식별자를 포함한다.

### 예시

```java
log.info("주문 생성 완료. orderId={}", orderId);
```

```java
log.error("결제 실패. orderId={}", orderId, e);
```

## 테스트 코드 작성

### 배경
- 테스트는 기능의 안정성을 보장하고 리팩토링에 대한 부담을 줄여준다.
- 테스트가 없는 코드는 변경 시 예상치 못한 사이드 이펙트를 유발할 수 있다.

### 원칙
- 비즈니스 로직은 가능한 한 테스트 코드를 작성한다.
- Happy Path 뿐 아니라 예외 케이스도 검증한다.
- 테스트는 구현이 아닌 동작(Behavior)을 검증한다.
- 테스트끼리 의존하지 않는다.
- 하나의 테스트는 하나의 시나리오만 검증한다.
- 테스트 이름만 보고도 어떤 상황을 검증하는지 이해할 수 있어야 한다.

### 예시

나쁜 예시

```java
@Test
void test1() {
}
```

좋은 예시

```java
@Test
void 탈퇴한_회원은_플랜을_생성할_수_없다() {
}
```

### 권장 패턴

```java
// given
User user = User.withdrawn();

// when
planService.create(user);

// then
assertThatThrownBy(...)
```

## 테스트 가능성(Testability)을 고려한 설계

### 배경
- 테스트하기 어려운 코드는 유지보수도 어렵다.
- 테스트를 쉽게 작성할 수 있는 구조는 자연스럽게 관심사가 분리된다.

### 원칙
- 외부 API, 시간, 파일 시스템 등의 의존성은 추상화한다.
- static 호출을 최소화한다.
- 서비스는 생성자를 통한 의존성 주입을 사용한다.

### 예시

나쁜 예시

```java
LocalDateTime now = LocalDateTime.now();
```

좋은 예시

```java
private final Clock clock;

LocalDateTime now = LocalDateTime.now(clock);
```

## 주석보다 코드로 의도를 표현한다

### 배경
- 주석은 코드 변경 시 함께 수정되지 않아 쉽게 오래된 정보가 된다.
- 좋은 이름과 구조는 주석보다 더 정확하게 의도를 전달한다.

### 원칙
- 코드만으로 의도를 표현할 수 있다면 주석을 작성하지 않는다.
- 변수명, 메서드명, 클래스명을 통해 의도를 드러낸다.
- "무엇을 하는지"보다 "왜 하는지"를 설명하는 주석을 작성한다.

### 예시

나쁜 예시

```java
// 유저를 조회한다.
User user = userRepository.findById(id);
```

좋은 예시

```java
User activeUser = findActiveUser(id);
```

## 주석 작성 기준

### 배경
- 메서드명만으로는 사용 목적이나 정책을 완전히 전달하기 어려운 경우가 있다.
- Controller, Service의 공개 메서드는 역할을 빠르게 파악할 수 있어야 한다.

### 원칙
- 공개(Public) 메서드에는 역할과 주요 정책을 설명하는 주석을 작성한다
- 코드를 읽지 않아도 해당 메서드의 책임을 이해할 수 있어야 한다.
- private 메서드에는 특별한 이유가 없다면 주석을 작성하지 않는다.

### 좋은 예시

```java
/**
 * 전용면적 목록 조회
 * 건축물대장 API 실패 시 fallback으로 사용한다.
 * 건물명 매칭을 우선 수행하고 실패 시 동 단위로 조회한다.
 */
public List<Double> fetchAvailableAreas(...) {
}
```

```java
/**
 * 기사 수정 잠금을 획득한다.
 * 이미 다른 사용자가 수정 중인 경우 예외를 발생시킨다.
 */
public void acquireEditLock(...) {
}
```

```java
// 국토교통부 API는 동일 요청을 1분 내 5회 이상 호출하면 차단되므로 캐싱 적용
```

```java
// 기존 데이터와의 호환성을 위해 null을 허용
```


### 나쁜 예시

```java
/**
 * 유저를 조회한다.
 */
public User getUser(Long id) {
}
```

```java
/**
 * 리스트를 생성한다.
 */
List<String> list = new ArrayList<>();
```

### 주석 작성 기준

다음 중 하나 이상에 해당하면 작성할 수 있다.

- 공개 API의 역할 설명
- 비즈니스 정책 설명
- 예외 발생 조건 설명
- fallback 정책 설명
- 외부 시스템과의 연동 정책 설명
- 외부 시스템 제약
- 성능 최적화 이유
- 기술적 우회 방법

다음은 작성하지 않는다.

- 코드 한 줄 설명
- 변수 설명
- 메서드 이름을 그대로 반복하는 설명

## TODO 주석은 기한 없이 남겨두지 않는다

### 배경
- TODO 주석은 시간이 지나면 방치되기 쉽다.
- 오래된 TODO는 오히려 혼란을 유발한다.

### 원칙
- TODO 작성 시 이유를 함께 작성한다.
- 담당자 또는 이슈 번호를 함께 기록한다.
- 해결 계획이 없는 TODO는 작성하지 않는다.

### 예시

좋은 예시

```java
// TODO(#123): 국토부 API 응답 포맷 변경 후 DTO 수정 필요
```

나쁜 예시

```java
// TODO 나중에 수정
```

## 테스트 코드도 운영 코드와 동일한 품질 기준을 적용한다

### 배경
- 테스트 코드도 유지보수 대상이다.
- 테스트 코드가 복잡하면 신뢰하기 어려워진다.

### 원칙
- 테스트 코드도 중복을 최소화한다.
- 테스트 코드도 의미 있는 이름을 사용한다.
- 테스트 코드도 리팩토링한다.
- 테스트를 통과시키기 위한 억지 구현을 하지 않는다.

### 예시

나쁜 예시

```java
assertThat(result.getValue()).isEqualTo("SUCCESS");
assertThat(result.getCode()).isEqualTo(200);
assertThat(result.getStatus()).isEqualTo("OK");
...
```

좋은 예시

```java
assertThat(result).isEqualTo(expectedResult);
```
